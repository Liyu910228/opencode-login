# 项目上下文采集

当需求要在某个既有项目里生成页面、演示页或实现方案时使用本文件。目标项目是风格和实现习惯的取证来源，不是本 skill 的落点。

## 采集顺序

1. **确认目标项目路径与运行环境**
   - 记录项目路径、用户希望对齐的页面类型、最终产物放置位置。
   - 如果 skill 运行在 opencode / harness 中，产物默认写在 harness 工作区；不要把 skill 自身复制到业务项目。

2. **读取项目代理规则**
   - 如果存在 `AGENTS.md`，只提取和当前任务有关的约束：沟通语言、实现前置检查、验证方式、提交/测试要求、项目特有工作流。
   - 不要把 commit、GitNexus、测试等工程约束硬塞进纯 demo 产物；只在需要真实代码实现时使用。

3. **读取项目本地 skills**
   - 扫描 `skills/*/SKILL.md` 或 `.agents/skills/*/SKILL.md`。
   - 摘要成"会影响页面实现的项目经验"，例如 scroll-view 高度坑、swiper sticky 清理、百分比字段不要乘 100、长名称独占行。
   - 按页面特征加载相关 skill；不要无差别读取全部长文。

4. **寻找同类真实页面**
   - 优先找同名页面、同业务域页面、同交互结构页面、同数据展示页面。
   - 对每个候选记录：路径、页面用途、关键结构、可复用视觉模式、不能照搬的差异。
   - 如果找不到同类页面，明确写"无同类基线"，再退回通用 Snow Mobile 规范。

5. **对比真实页面与生成目标**
   - 抽取单位、颜色、字体、间距、圆角、阴影、表格、tab、筛选、图表、吸顶、swiper、空状态等模式。
   - 对于 `1 到 N` 需求，真实代码优先于旧文档和通用规范。

## 输出到 requirement.md

在 `requirement.md` 的现有流程分析中加入：

- 目标项目路径
- 相关项目规则摘要
- 相关本地 skills 摘要
- 同类页面候选与采用理由
- 本次需求需要继承的结构/视觉/实现习惯
- 仍待确认或无法验证的项目上下文

## 禁止事项

- 禁止只看 PRD 就直接生成页面。
- 禁止把通用 dashboard 模板当作项目真实风格。
- 禁止把目标项目的全部规则原样复制进本 skill。
- 禁止把页面是否"像项目"只理解成功能是否完整；必须检查观感、结构模式和实现习惯。
