# 统一需求分析规则

这是 `zfl-requirement` 的统一需求分析规则。目标不是快速写文档，而是先识别关键缺口，通过少量高价值追问，把模糊需求稳定还原为可落地的 `requirement.md`。

## 1. 适用范围

所有类型的需求分析都使用本规则，包括但不限于：

- 移动端/PC 数据看板、经营看板、销售看板、销量快报
- KPI 指标卡、趋势图、结构占比图、排名表、组织明细表
- 按时间、组织、品类、品牌、品种、区域切换的数据分析页面
- 支持 Tab、侧边导航、连续浏览、下钻的数据页面
- 流程审批、交易、表单录入、配置后台、列表详情、工作台
- AI 生成、AI 辅助决策、AI 检索、AI 总结、AI 自动化处理类需求

即使核心不是数据看板，也不能回退到旧的通用需求分析；要按本规则的提问节奏、确认方式和写作要求执行。遇到非看板需求时，把“看板、Tab、指标、下钻”等词按业务场景映射为“页面、流程、对象、字段、操作、状态、权限、验收”。

## 2. 开始前先做三类判断

每次开始前必须判断：

| 判断项 | 说明 |
| --- | --- |
| 需求类型 | 数据看板 / 传统流程 / AI 需求 |
| 演进方式 | `0 到 1` 新建 / `1 到 N` 改造已有 |
| 终端形态 | 移动端 / PC Web / 大屏 / 多端复用 |

判断后先向用户简要说明判断结果。

如果是 `1 到 N`，优先读取旧页面、旧需求、代码、BI、Excel、截图或原型，先还原现状，再定义改动范围。

## 3. 优先识别五类高风险缺口

不要机械逐条提问，优先判断下面五类是否清楚。

### 3.1 目标与用户

必须明确：

- 谁看这个看板
- 看完后要做什么判断、跟进或汇报
- 当前通过什么方式看数据
- 首版最必须解决什么问题

### 3.2 结构、流程与导航

必须明确：

- 页面入口和页面名称
- 默认一级 Tab、默认二级主题、默认时间口径、默认组织范围
- 一级 Tab、二级主题、模块顺序
- 二级主题通过侧边导航、下拉、卡片入口还是连续浏览切换
- 哪些模块只在总览展示，哪些模块在单项主题展示
- 对流程类需求，必须明确主流程、分支流程、异常流程、触发入口、结束状态
- 对表单类需求，必须明确新建、编辑、查看、提交、撤回、审批、驳回等动作边界
- 对 AI 类需求，必须明确 AI 位于主流程、辅助流程还是后台自动处理

### 3.3 指标、对象与字段

必须明确：

- 核心指标、单位、时间口径
- 维度：组织、档次、品牌、品种、区域等
- 汇总卡片字段
- 图表类型、维度、指标、颜色含义
- 表格字段、排序、分组表头、横向滚动、是否下钻
- 衍生指标公式：同比增减、同比增幅、占比、占比差异、增量贡献
- 对流程类需求，必须明确业务对象、对象状态、字段、校验规则、状态流转条件
- 对表单类需求，必须明确字段名称、字段类型、必填规则、默认值、可编辑条件
- 对 AI 类需求，必须明确输入、输出、上下文来源、输出格式、人工兜底、可接受误差

### 3.4 权限、操作与下钻

必须明确：

- 有哪些角色或组织层级
- 不同角色默认看到什么范围
- 是否允许切换组织范围
- 哪些表格行可下钻
- 下钻到什么层级
- 下钻用底部弹窗、详情页、展开行还是二级页面
- 是否只下钻一级
- 不同角色可执行哪些操作、不可执行哪些操作
- 哪些字段、按钮、状态、审批节点或 AI 结果需要权限控制
- 权限是沿用现有系统，还是本次新增或调整

### 3.5 状态、异常与验收

必须明确：

- 加载、无数据、失败、无权限如何展示
- 表格字段过多时是否固定首列、横向滚动
- 本期不做什么
- 验收时逐条检查哪些默认状态、切换规则、字段、下钻和异常
- 对流程类需求，必须明确待处理、处理中、已完成、已取消、已驳回、失败等状态
- 对 AI 类需求，必须明确生成中、生成失败、低置信度、人工确认、重试、回滚等状态

## 4. 三条执行红线

### 4.1 不确定不写死

未确认的信息不能写成确定需求。

| 信息状态 | 写法 |
| --- | --- |
| 已确认 | 写成正式需求 |
| 高概率但未确认 | 写成“建议默认值”或“暂按……设计” |
| 会影响结构/字段/口径/权限 | 必须优先追问 |

禁止把未确认内容写成：

- `系统支持 xxx`
- `页面展示 xxx`
- `用户可通过 xxx 筛选`
- `表格字段为 xxx`

应写成：

- `待确认：xxx`
- `建议默认值：xxx`
- `暂按 xxx 设计，若实际为 yyy，则影响 zzz`

### 4.2 原文术语不能直接映射成组件或功能

原文出现以下词时，必须先判断含义：

- 筛选按钮
- 档次筛选
- 品种筛选
- 导航
- 菜单
- 入口
- Tab
- 切换
- 分组
- 维度
- 标题
- Banner 标题
- 新增
- 编辑
- 审批
- 同步
- AI 生成
- 自动处理

必须判断它是：

| 可能含义 | 处理 |
| --- | --- |
| 真实筛选条件 | 确认筛选项、默认值、刷新范围 |
| 二级主题导航 | 写入导航结构，不写成筛选功能 |
| 页面标题说明 | 写入标题规则 |
| 维度分组 | 写入指标维度或表格分组 |
| 操作文案 | 确认是静态提示、点击切换还是手势切换 |
| 业务动作 | 确认触发条件、处理对象、前置条件、成功状态、失败状态 |
| AI 能力 | 确认模型输入、输出格式、人工确认、失败兜底和评估方式 |

不确定时先问：

> 原文中的“xxx”是实际可点击筛选按钮，还是侧边导航/主题切换说明？

### 4.3 卡片、表格、表单和流程必须追问到字段级

只要出现“汇总卡片、指标卡、表格、排名表、明细表、结构表、表现表、表单、审批单、详情页、配置项、AI 输出结果”，必须追问到字段级。

汇总卡片至少确认：

- 主指标
- 单位
- 字段名是否随时间口径变化，例如当日销量、当月销量、当年销量
- 是否展示同期、同比增减、同比增幅
- 是否按多个维度分别展示

表格至少确认：

- 表格名称
- 组织页签
- 行维度
- 字段列
- 是否有分组表头
- 分组表头下有哪些指标
- 排序规则
- 是否可切换排序
- 是否下钻
- 下钻层级
- 横向滚动与固定列规则

表单至少确认：

- 表单名称
- 适用角色
- 字段列
- 字段类型
- 必填规则
- 默认值
- 校验规则
- 可编辑条件
- 提交后状态
- 提交失败处理

流程至少确认：

- 流程名称
- 触发入口
- 参与角色
- 主流程步骤
- 分支条件
- 每一步输入输出
- 状态流转
- 异常处理
- 撤回、驳回、重试、取消规则

AI 输出至少确认：

- 输入来源
- 输出字段
- 输出格式
- 是否允许用户编辑
- 低置信度或失败时如何处理
- 人工确认后如何进入后续流程
- 如何验收 AI 效果

不同主题不能默认复用同一套模块。即使看起来相似，也要逐个确认。

## 5. 提问策略

执行流程：

1. 先用 3-5 条复述已理解的信息
2. 说明三类判断结果：需求类型、演进方式、终端形态
3. 标出当前最大风险缺口
4. 每轮只问 3-6 个最高优先级问题
5. 每轮回答后更新：
   - 已确认
   - 仍不清楚
   - 建议默认假设
   - 待拍板
6. 当关键结构和口径足够稳定后，再产出 `requirement.md`

不要一次性问几十个问题。问题必须能直接减少范围、口径、字段、权限或验收上的返工。

面对只给出功能名的模糊需求，例如“做一个销量快报”，优先问：

- 谁看，默认看全国还是自己负责组织
- 第一屏最重要的 1-3 个指标是什么
- 需要按哪些大类切换
- 默认时间口径是什么
- 是否需要组织排名和下钻

面对非看板模糊需求，也按同样节奏先问会影响范围和验收的问题。例如：

- 做给谁用，当前这件事现在怎么处理
- 第一版最必须跑通的主流程是什么
- 入口在哪里，默认进入后看到什么
- 关键对象和字段有哪些
- 有哪些角色、权限和状态流转
- 失败、无权限、无数据或 AI 结果不可信时怎么处理

## 6. 开始写 `requirement.md` 的条件

满足以下条件后再开始写：

- 页面名称、核心用户、业务目标基本明确
- 默认进入状态明确
- 一级视角、二级主题、模块结构能列清楚
- 关键卡片、图表、表格能落到字段级，或已标注待确认
- 指标公式、单位、时间口径基本明确
- 权限范围和下钻层级基本明确
- 异常状态、本期不做范围、验收标准可写
- 对流程、表单、AI 类需求，主流程、关键字段、状态流转、权限边界、失败兜底和验收方式基本明确

如果仍有关键缺口，不要假装完整；继续追问或显式标为“待确认”，并说明影响范围。

## 7. `requirement.md` 建议结构

输出文档建议使用以下结构，按实际需求删减。

```markdown
# [需求名称]需求文档

## 1. 文档信息
- 需求名称：
- 页面类型：
- 适用端：
- 核心用户：
- 版本：
- 来源：

## 2. 背景与目标
### 2.1 业务背景
### 2.2 业务目标
### 2.3 默认进入状态

## 3. 页面信息架构
### 3.1 页面整体结构
| 区域 | 内容 | 说明 |
| --- | --- | --- |
### 3.2 一级视角与二级主题
### 3.3 导航与切换规则
### 3.4 连续浏览 / 底部切换规则（如有）

## 4. 功能需求
按“一级视角 - 二级主题 - 模块”拆分。
每个模块说明：
- 模块名称
- 展示位置
- 字段 / 指标
- 图表或表格形式
- 排序 / 下钻 / 横向滚动
- 受时间、组织、权限影响的规则

## 5. 数据与计算口径
- 通用字段
- 指标公式
- 单位与精度
- 排名规则
- 空值、负值、同期为 0 规则
- 数据来源与刷新规则

## 6. 权限与下钻
- 角色 / 组织层级
- 默认数据范围
- 可切换范围
- 下钻入口
- 下钻层级
- 下钻展示方式

## 7. 状态与异常
- 加载中
- 无数据
- 加载失败
- 无权限
- 表格溢出
- 下钻失败

## 8. 非功能与本期不做
- 适配范围
- 性能要求
- 本期不做范围

## 9. 埋点建议
- 页面访问
- Tab 切换
- 主题切换
- 时间切换
- 组织切换
- 下钻点击

## 10. 验收标准
逐条写成可测试项。

## 11. 待确认问题
| 问题 | 影响范围 | 建议处理 |
| --- | --- | --- |
```

## 8. 写作要求

生成文档时遵守：

- 用业务语言，不写空泛描述
- 按主题逐个拆，不跨主题泛化
- 卡片、图表、表格必须落到字段、口径、排序、下钻
- 未确认内容必须显式标注
- 默认假设必须写明影响范围
- 验收标准必须可测试
- 用户每次补充后，同步检查功能需求、口径、权限、下钻、验收是否需要更新
