# 页面规则

## 产品气质

Snow Mobile 页面是内部业务 APP 页面，应体现高效、冷静、数据清晰和可操作。优先保证扫读效率、层级清楚、任务完成路径短、交互模式稳定。

避免：

- 营销式大首屏或英雄区；
- 用装饰插画作为页面主体结构；
- 每个模块都做成漂浮卡片，导致页面过碎；
- 标题过大，压缩业务信息密度；
- 不对应明确业务需求的一次性视觉变体。

## 移动端页面结构

使用稳定、可预期的页面结构：

1. APP 导航/顶部栏：页面标题、必要的返回操作，以及最多一个页面级操作。
2. 关键信息摘要：最重要的状态、金额、数量或实体身份。
3. 主体内容：业务字段分组、表格/列表行、图表、表单输入或流程步骤。
4. 场景化操作：主操作优先，次操作靠近但视觉弱化。
5. 反馈状态：按需要补齐加载、空、错误、禁用、已提交、无权限等状态。

### 页面骨架

移动端页面优先按以下骨架组织：

| 区域 | 是否必需 | 用法 |
| --- | --- | --- |
| 顶部导航 | 必需 | 承载页面标题、返回、关闭、搜索或页面级操作 |
| 页面内容 | 必需 | 承载业务信息、表单、列表、图表、流程和状态 |
| 底部操作 | 按需 | 承载提交、确认、审批等高优先级动作 |
| 底部导航 | 按需 | 承载 APP 一级模块切换，不用于页面内状态切换 |
| 浮层反馈 | 按需 | 承载 Toast、弹窗、确认、加载等临时反馈 |

页面内容是必需区域；其他区域只有在任务、导航或反馈需要时出现。

### 顶部导航

- 标题必须准确表达当前页面或当前对象。
- 返回操作只返回上一级，不承载业务取消逻辑。
- 右侧页面级操作最多保留一个高频入口；更多操作使用更多菜单或放入内容区。
- 搜索页、筛选页、选择页可以用搜索框或选择器替代普通标题，但仍要保留清晰的返回路径。
- 顶部导航不放复杂说明文案，说明应进入内容区。

### 内容区层级

内容区按“先判断、再处理、后查看详情”的顺序组织：

1. 当前对象或当前范围；
2. 关键状态、关键指标或关键结论；
3. 当前用户最可能执行的操作；
4. 主体数据、字段、列表、图表或表单；
5. 次要信息、日志、附件、说明。

当页面是纯录入或纯选择流程时，可以把表单/选择列表提前到关键状态之前。

### 筛选与切换

- 页面内同级内容切换使用选项卡或分段器。
- 跨页面或跨一级模块切换使用导航，不使用选项卡。
- 列表筛选优先放在列表上方；筛选项多时使用筛选面板。
- 日期、组织、区域、门店等范围选择应靠近指标或列表标题。
- 搜索、筛选、排序、选项卡状态应相互独立，避免切换一个控件导致其他条件被意外清空。

### 内容承载

- 列表页优先使用列表承载重复对象；每行突出标题、状态和 1-3 个关键字段。
- 卡片用于信息密度更高、需要分组或突出操作的对象；不要把所有普通列表都做成卡片。
- 表单页按业务分组组织字段；错误提示紧跟字段，不集中堆在页面顶部。
- 看板页先展示总览指标，再展示趋势、排行、异常和下钻入口。
- 详情页优先展示状态和关键字段，完整字段放入分组详情。

### 底部操作

- 高优先级提交、确认、审批、保存使用底部固定操作。
- 底部操作区只保留 1 个主操作；可搭配 1 个次操作。
- 底部操作不得遮挡内容，页面底部需要预留安全距离。
- 危险操作不要与正向主操作并列成同等视觉重量。
- 只读详情页不强行添加底部操作。

### 反馈层级

- 轻量成功、失败、复制、保存反馈使用 Toast。
- 需要用户确认的高风险操作使用弹窗。
- 表单校验错误优先就近提示。
- 页面级错误展示可恢复动作，如重试、返回、联系管理员。
- 加载态要避免页面跳动；列表加载更多和页面首次加载要区分。

### 空状态与异常状态

- 空状态说明当前范围没有数据，并提供下一步动作。
- 无权限状态说明权限原因和可申请路径。
- 网络错误、服务错误、数据异常要给出重试或退出路径。
- 没有业务动作时，不要强行放主按钮。

### 页面导航关系

- 一级模块切换使用底部导航或应用主导航。
- 同一对象的子内容切换使用选项卡。
- 从列表进入详情，详情页保留返回列表路径。
- 从详情进入编辑，编辑完成后返回详情或结果页，不跳到无关页面。
- 流程型页面要明确上一步、下一步、提交成功和提交失败去向。

## 从 PRD 提取页面信息

根据 PRD 生成页面时，先提取：

- 页面名称和业务目标；
- 目标用户角色；
- 页面入口、出口和跳转目标；
- 必填数据字段；
- 操作项及优先级；
- 状态机或业务状态值；
- 校验规则；
- 权限规则；
- 空、错误、加载状态；
- 看板指标或埋点指标。

如果 PRD 有歧义，基于内部业务 APP 的常见逻辑做合理推断，并简短标注假设。

## 常见页面模式

### 详情页

用于订单、发票、门店、审批、拜访、客户、库存、任务等详情。

推荐顺序：

1. 实体身份和当前状态；
2. 关键数字或截止时间；
3. 主操作；
4. 分组详情；
5. 流程、日志或附件。

### 看板页

用于移动端经营或运营看板。

看板页、数据看板、月报、日报、经营分析、库存分析、销量分析、风险监控页面的硬规则统一维护在 `dashboard-pattern.md`。生成、调整或检查此类页面时，必须以 `dashboard-pattern.md` 为唯一事实来源。

推荐顺序：

1. 日期或范围选择；
2. 2-4 个顶部核心 KPI；
3. 趋势或排行模块；
4. 异常/提醒列表；
5. 下钻入口。

图表要简单，保证小屏可读。

### 列表页

用于可搜索、可筛选的业务集合。

推荐顺序：

1. 标题、搜索、筛选；
2. 当业务有强状态分组时使用选项卡；
3. 列表行展示状态和关键元数据；
4. 分页或加载更多；
5. 空状态和重试。

### 表单页

用于新建、编辑、提交或审批录入。

推荐顺序：

1. 字段分组；
2. 校验提示靠近对应字段；
3. 关键提交流程使用底部固定操作；
4. 提交后给出确认或结果状态。

## 布局判断

- 移动端优先使用单列流式布局，只有紧凑指标对比时再使用多列。
- 只有当分组能提升扫读效率时才增加区块。
- 主操作要清楚、可达、位置稳定。
- 标签文案使用 PRD 或业务中的真实术语。
- 示例内容优先使用真实业务数据，不使用无意义占位文。
- 确保文字不重叠，关键数字和状态不被截断或遮挡。
