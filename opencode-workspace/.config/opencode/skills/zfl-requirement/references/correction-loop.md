# 纠正与反写

用户看完 `demo.html` 后大概率会提出不满意或纠正意见。纠正阶段不能只修表面页面，也不能用户说一点就立刻改一点；必须像需求整理阶段一样，先归并反馈、判断影响面、确认本轮修正范围，再把确认后的修正反写回上游产物。

## 反馈分层

| 反馈类型 | 示例 | 必改产物 |
|----------|------|----------|
| 业务理解错 | "这个不是按品牌看，是按档次看" | `requirement.md`、`reqdoc.md`、`demo.html` |
| 字段/口径错 | "同比增幅不是乘 100 后再展示" | `reqdoc.md`、`demo.html` |
| 权限/角色错 | "业务部不能看全国数据" | `requirement.md`、`reqdoc.md`、`demo.html` |
| 页面结构错 | "不应该堆卡片，要像销量快报那种分屏" | `reqdoc.md`、`demo.html` |
| 风格不像项目 | "看起来不像这个项目里的页面" | `reqdoc.md`、snow-design-mobile 项目覆盖规则、`demo.html` |
| 当前 demo bug | "按钮点不开" | `demo.html` |

## 执行顺序

1. 收集用户反馈，先不急着改。
2. 归并反馈为待修正清单，去重、合并同类项，区分事实错误、偏好表达和新需求。
3. 判断每条反馈的层级和影响面，并列出可能牵连的页面、字段、权限、口径、交互、文档和设计规则。
4. 给出本轮建议修正批次，向用户确认；用户确认前不要开始修改。
5. 用户确认后，先改上游：`requirement.md` / `reqdoc.md`。
6. 再改 `demo.html`。
7. 重新检查字段、文案、状态、视觉结构是否一致。
8. 重新给预览链接或可打开路径。

## 确认格式

纠正前用简短清单向用户确认，避免越改越乱：

```markdown
我先把你的反馈归并成这一轮修正：

1. {{修正点}}  
   - 类型：业务理解 / 字段口径 / 页面结构 / 风格规则 / demo bug / 新需求
   - 影响：{{会影响哪些文档、页面、字段或规则}}
   - 建议：本轮处理 / 放入待确认 / 作为新需求另起一轮

本轮我建议先处理：{{范围}}。
暂不处理：{{范围与原因}}。
确认后我会同步更新 requirement.md / reqdoc.md / demo.html 中受影响部分。
```

如果反馈明显互相冲突，或会扩大需求范围，必须停在确认清单阶段。不要自行选择一个方向直接改。

## 反写规则

- 用户确认后的修正是新事实，不能只写在聊天里。
- 如果修正影响需求范围、业务规则、权限或验收，必须写回 `reqdoc.md`。
- 如果修正影响页面骨架、模块顺序、同类页面基线或视觉风格，必须写回 `reqdoc.md` 的页面与视图、交互步骤、关键状态、验收标准等受影响章节。
- 如果修正暴露出通用设计 skill 的缺口，补充到 snow-design-mobile 对应 reference，而不是只修当前 demo。
- 如果只是一次性展示数据或点击 bug，不要污染上游规则。
- 如果反馈实际是新增需求，不要混入当前纠正；单独列为下一轮需求，等用户确认范围后再进入需求整理。

## 不要这样做

- 不要只改 `demo.html` 让当前截图过关。
- 不要用户说一点就改一点，导致文档、规格和 demo 逐步发散。
- 不要把用户反馈当成新的设计偏好无限追问。
- 不要让用户在纠正阶段承担组件、颜色、布局等专业决策。
- 不要在文档和 demo 之间留下字段名称、口径、状态不一致。
